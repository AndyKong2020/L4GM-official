## 1. 技术栈梳理
- 主语言:Python。
- ML 框架:PyTorch + torch_npu。L4GM 主体为 U-Net + Gaussian renderer;多视图生成使用 ImageDream pipeline。
- CUDA 依赖(必需/可选):xFormers 与 gsplat 为 CUDA 生态依赖。xFormers 已回退 PyTorch attention;CUDA `gsplat` 使用简化 NPU renderer fallback 替代。
- 自定义核(.cu / C++ 扩展):仓内推理主路径无必须编译的 C++/CUDA 扩展;官方视觉质量依赖的 `gsplat.rendering.rasterization` 不可在 Ascend 上直接使用。
- 第三方库:diffusers、transformers、safetensors、torchvision、opencv、imageio、kiui、tyro。
- 模型权重 / 来源:`recon.safetensors`、`interp.safetensors`、ImageDream `image_encoder/text_encoder/unet/vae`,部署时放入 `pretrained/`。

## 2. 部署步骤
- [x] 依赖安装:Python 3.11 环境,安装 torch_npu、torchvision、diffusers、transformers、safetensors、opencv、imageio、kiui、tyro。
- [x] 编译 / 构建:无必需 C++/CUDA 扩展;NPU fallback 路径通过 `python -m py_compile`。
- [x] 权重获取:权重放入 `pretrained/`,不纳入 git。
- [x] NPU 适配改动(device、torch_npu、禁用 CUDA 核等):
  - `core/device.py` 统一 NPU 设备、cache、autocast、empty cache 与 CLIP tensor materialize。
  - `infer_3d.py`、`infer_4d.py`、`main.py` 替换 CUDA-only device/autocast/cache,推理阶段禁用 LPIPS 下载。
  - `mvdream/pipeline_mvdream.py` 在迁移 ImageDream encoder 前 materialize CPU tensor,规避 checkpoint-backed tensor `.to(npu)` 卡死。
  - `mvdream/mv_unet.py` 缺 xFormers 时走 PyTorch attention fallback。
  - `core/gs.py` 缺 CUDA gsplat 时走 NPU 功能 fallback rasterization。
- 命令:
```bash
source scripts/npu_env.sh

python infer_3d.py big \
  --workspace results_npu_materialized \
  --resume pretrained/recon.safetensors \
  --num_frames 1 \
  --test_path data_test/otter-on-surfboard_fg.mp4

python infer_4d.py big \
  --workspace results_npu_materialized \
  --resume pretrained/recon.safetensors \
  --interpresume pretrained/interp.safetensors \
  --num_frames 10 \
  --test_path data_test/otter-on-surfboard_fg.mp4
```

## 4. NPU 亲和性
口径:Ascend 950PR、3510 架构、FP16/autocast、单进程单卡推理。平衡点是计算与访存瓶颈分界,950PR FP16 Cube 约 270 FLOP/Byte,Vector FP16 约 34 FLOP/Byte。

| 指标 | 数值 |
|---|---|
| 能否在 NPU 跑通 | 3D 跑通;4D `num_frames=4/8/10` 跑通;`num_frames=11/12/16` OOM |
| NPU 利用率 (npu-smi) | `num_frames=8` 采样:NPU Util 99-100%,AICore 95-100%,AIVector 65-84%,AICube 14-27%,HBM Usage 54% |
| HBM 占用 | 3D 7.38/9.52 GiB;4D 10 帧 68.96/92.49 GiB;11 帧开始 OOM |
| 关键算子是否回退 CPU | 主干未观察到 CPU fallback;CLIP 加载期需 CPU materialize;CUDA gsplat 被 fallback 替代 |
| 性能(吞吐/时延) | ImageDream 30-step 约 17.2 it/s;3D 8.94s;4D 4/8/10 帧为 64.86/75.29/78.83s |

**计算分布**(实测,Level1 + PipeUtilization):

| 单元 | 主要算子 | 占比 / 现象 | 说明 |
|---|---|---:|---|
| 算力(Cube) | BatchMatMul、Conv2D | 约 21% | BatchMatMul 12.2%,Conv2D 合计约 9.2%;不是纯 Cube 饱和 |
| 向量(Vector) | Cast、Softmax、GroupNormSilu、Swish | 约 73% | Cast 43.4%,Softmax 25.4%,Vector/格式转换压力高 |
| 搬运(MTE/FixPipe) | attention matrix materialize、Cast、cat/repeat/contiguous | 高 | 10 帧 reserved 92.49 GiB,11 帧以上 HBM 不可承受 |
| 通信(communication) | 无 | 0 | 单卡,step trace communication 为 0 |
| 调度(host/head) | 4D 分段循环、ImageDream diffusion、视频编码 | 中 | 计算段 NPU Util 接近满载,端到端仍受 Python/IO/编码影响 |

**热点算子明细**(`forward_gaussians`,3 step):

| 算子 | 占比 |
|---|--:|
| Cast | 43.4% |
| Softmax | 25.4% |
| BatchMatMul | 12.2% |
| Conv2D | 约 9.2% |
| GroupNormSilu | 2.8% |
| Swish | 2.1% |
| Transpose | 1.6% |

**renderer fallback 热点**(单视角,3 step):Slice 10.7%、NonZero 8.9%、Mul 8.9%、Pack 7.1%、Cast 7.0%、ClipByValueV2 6.9%、RealDiv 6.6%、GatherV2 5.5%、ScatterAdd/ScatterElements 约 5.6%。该路径为 Vector/不规则负载,亲和低于规整 Conv/GEMM。

- 算子回退清单:
  - xFormers CUDA attention 不可用 → PyTorch attention fallback。
  - CUDA `gsplat` 不可用 → 简化 NPU renderer fallback。
  - CLIP checkpoint-backed tensor 直接迁移阻塞 → CPU materialize 后迁移。
- profiler 摘要:仓内无 `enable_profiler` 内置路径,采用注入采集。配置为 `ExperimentalConfig(Level1 + PipeUtilization)`,profile 外预热,schedule `active=3`,额外 `prof.step()` 收尾。`forward_gaussians` 产物 `kernel_details.csv` 3906 行/49 列;renderer 产物 222 行/49 列;trace 与 op/api/operator/step CSV 均生成。

## 5. 阻塞项
| 阻塞点 | 原因 | 是否硬阻塞 | CANN/AscendC 替代方案 | 兜底 |
|---|---|---|---|---|
| ImageDream CLIP encoder `.to(npu)` 卡死 | checkpoint-backed CPU tensor 与 torch_npu copy path 不兼容 | 否,已规避 | 加载后 materialize 为普通 contiguous tensor | 保持 `materialize_cpu_module_tensors()` |
| 4D 高帧数 OOM | PyTorch attention fallback 显式构造 `[N,N]` attention matrix,softmax workspace 随帧数暴涨 | 是,影响 11 帧以上和 README 默认 16 帧 | online softmax、chunk attention、NPU SDPA/AscendC memory-efficient attention | 以 `num_frames<=10` 复现 |
| CUDA `gsplat` 不可用 | 官方 Gaussian rasterizer 依赖 CUDA 扩展 | 是,影响官方视觉质量等价 | NPU tile binning + sort + alpha blend | 简化 fallback 只保证功能产物 |

## 6. 结论
- 运行方案(NPU / NPU+CPU / CPU):NPU 为主。3D 与 4D 10 帧以内已跑通;加载期 CLIP materialize 使用 CPU 预处理;renderer 使用 NPU 功能 fallback。
- 亲和性:Conv/MatMul 可用,但主要瓶颈在 Cast/Softmax/attention materialize 与 renderer 不规则 Vector 路径;高帧数 4D 受 HBM 限制。
- 适配状态:可作为 NPU 功能复现;实测完整视频上限为 10 帧;默认 16 帧和 CUDA gsplat 视觉质量等价未完成。
- 风险与优化方向:实现 memory-efficient attention 以恢复 11/12/16 帧;替换简化 renderer 为 NPU 版 Gaussian rasterizer;性能优化阶段再扩展 ImageDream diffusion 和端到端 profiler。
